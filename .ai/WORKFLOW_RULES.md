# 개발 워크플로우 규칙

## 1) 순서 강제
아래 순서를 반드시 지킨다.

1. 요청 수신 직후 아래 4개만 출력한다.
   - (1) Request Summary
   - (2) Plan
   - (3) Risks
   - (4) Approval Needed
2. 사용자 승인 전에는 실제 실행을 금지한다.
   - 파일 생성/수정
   - 커밋
   - PR 생성
   - 외부 시스템 변경
3. 승인되면 (5) Checklist를 작성한다.
4. Checklist 확정 후 GitHub Issue를 생성한다.
5. Issue 번호로 브랜치를 생성한다.
   - 규칙: `feature/<slug>-<issueNumber>`
6. 작업을 수행한다.
7. 변경 요약과 검증 결과를 먼저 보고하고 커밋 승인을 받는다.
8. 승인되면 커밋한다.
9. PR을 생성한다.

## 2) 출력 포맷
승인 전 출력은 아래 4개 섹션만 사용한다.

### (1) Request Summary
- 요청 요약 1~3줄

### (2) Plan
- 실행 계획 2~5개

### (3) Risks
- 예상 리스크와 대응

### (4) Approval Needed
- `yes/no`로 답할 수 있는 승인 질문 1개

승인 후 출력은 아래 섹션을 사용한다.

### (5) Checklist
- 확정된 TODO 목록

### (6) Execution Log
- 핵심 진행 로그만 간단히 기록

### (7) Verification
- 실행한 검증 명령과 결과 요약

### (8) PR Notes
- PR 본문에 들어갈 핵심 요약

## 3) 위험 작업 승인 규칙
아래 작업은 명시적 승인 없이는 절대 실행하지 않는다.

- 배포
- DB 스키마 변경
- 마이그레이션
- 데이터 변경/삭제
- 복구가 어려운 운영성 작업

## 4) 커밋/PR 작성 규칙
- 제목/본문은 짧고 사실 중심으로 작성한다.
- 군더더기 문구를 넣지 않는다.
- 이모지를 사용하지 않는다.
- 과도한 서술을 하지 않는다.
